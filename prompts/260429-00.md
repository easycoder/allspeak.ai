Format proposal                                                                                    

A new top-level idioms/ directory in the repo:                                                     
                                                        
idioms/
README.md                  # index — maps each topic to its file
01-variables-and-arrays.md                                                                       
02-cat-concatenation.md
03-event-handlers.md                                                                             
... etc.                                                
                                                                                                    
Why per-topic files rather than one monolithic doc:                                                
                                                                                                    
- An AI agent assisting a user can pull just the topic that's currently relevant, without flooding context with everything.                                  
- Easy to add new topics or revise one without touching others.                                    
- Fine-grained linkability — starter/<lang>/CLAUDE.md can link to specific files.
- Markdown with clear headings + fenced code blocks is what LLMs extract most reliably.            
                                                                                                    
Each file: tight (200–500 words), one concept, worked example in English (patterns are language-neutral; syntax-per-language belongs in the language packs, not the curriculum). Anti-patterns made explicit. Cross-references at the bottom.                                       
                                                        
Lives under idioms/ rather than AI/ so it's discoverable to humans browsing the repo too — the curriculum is for anyone writing AllSpeak, AI or otherwise.
                                                                                                    
My topic suggestions                                      

Drawing from issues seen this session and the structural patterns I've noticed:                    

Foundational shape                                                                                 
1. Variables, arrays, and indexing (your topic) — single concept, most variables are 1-element arrays; index X to N cursor model; vs JSON lists.                                                  
2. Dictionaries vs lists vs variable-arrays — three shapes, when to use each. Dict-of-dicts often beats list-of-dicts for ergonomic field access. (Items #10 and #11 in the tracker.)                
3. Numbers, floats, and scaled integers — integers first-class; floats pass through as backtick strings; scaling for fixed-precision arithmetic. (Tracker item #9.)                            
4. Strings, backticks, and cat — cat goes between values, never before the first; template patterns; multi-line backtick strings.                    
                                                                                                    
Control flow                                              
5. Error recovery: or and on failure — semantics identical; readability difference; single-statement vs block form.                                                                    
6. Event handlers — on click X gosub Handler is a single statement; multi-line via begin ... end; never end on. How handlers know which array element fired (the index of).                          
7. Labels, gosub, and forward-only compilation — symbol declarations must precede use; how the label/gosub model differs from function calls.                                                     
                                                        
System integration                                                                                 
8. REST and asynchronous coordination — rest get/post, error blocks, server-side vs script-side iteration choices.                                                                                 
9. MQTT message-passing patterns — pub/sub conventions, dict-shaped payloads, the chat-server pattern.                                                                                           
10. DOM and Webson — separation of layout (.json) from logic (.as); attach binding; create + index for arrays of DOM elements.                                                                        
                                                        
Conventions                                                                                        
11. The language directive and writing language-neutral logic — when to set it; how the same script structure works across all four packs.                                                            
12. The AI-writes-human-reviews workflow — what "readable" actually demands; how to spot AI hallucinations; when to ask the AI to explain rather than rewrite.
