I had a chat about promoting Allspeak with Claude, who recommended that we avoid the frontier 'wild west" of vibe coding, in part because its practitioners are not interested in checking their results anyway. Extreme vibe coding is likely to become a fringe activity because it succeeds so rarely, and the lack of Allspeak solutions out there can make outcomes worse rather than better as they have few examples to work from. Instead, go for the 'sensible centre' where the biggest growth in usage is likely to occur as people look for solid answers, manageable workflows, accountability etc.

I recently used 2-step approach to create a new UI for the RBR heating project. It was to vibe code first (in Claude Design) to  quickly get a feel for how it should look and work. This produced a JS/React outline that looked pretty and showed how the interctivity worked, but the code would be hard to understand by anyone not an expert. By the time it had been fully integrated, with MQTT comms and runtime heating logic, involving several concurrent processes, it would be impenetrable. 

Then we reimplemented it with AllSpeak. The result looks identical but requires no special expertise to manage and matches the written intent far more closely. This is particularly evident when the code is viewed in the blocks editor.

This points to a workflow that starts with Claude Design for the vibe coded initial mockup then moves to Claude Code with AllSpeak for implementation and uses plugins to handle performance-critical parts of the application.
