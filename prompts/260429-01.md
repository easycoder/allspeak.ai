# Curriculum topics

## Structure
AllSpeak is composed of domain modules, where each domain contributes part of the overall language vocabulary and syntax. A number of domains are bundled together as the standard product; others can be added as plugins. The compiler operates by testing each domain in turn to see if it can handle a given script construct. If it can, the result of the test goes into a program array. If not, the compiler passes to the next domain until either compilation succeeds or none are able to handle the construct, which signifies an error.

## Symbols
AllSpeak uses just 4 symbols:

Comments are introduced by ! at any point in a line (except inside quoted values) and continue to the end of that line.
Style: There should be enough comments to identify major functional blocks (don't rely on variable names) and on the same line as code in less frequent cases where an explanation is considered valuable.

Constant text strings are delimited by backticks

Labels are identified by a trailing colon

Negative numbers are prefixed by -

Any other non-alphanumeric symbol outside a quoted string is an error.

## Indenting
Labels start at the left-hand margin. Code starts one tab position in, unless it is between begin...end, in which case it is further indented in the same way as other languages.

## Variables
A variable is something that holds a value or represents an entity of some kind, such as a file or a DOM element. Each domain has its own set of variable types with accompanying syntax and vocabulary.
All variable names start with a capital. Camel case is fine as long as the first letter is uppercase.
Variable names are chosen to clearly represent the values they contain.
All variables are global. The only way to achieve private variables is indirect, by running a child module (qv).
Variables should be grouped by type, and in functional rather than alphabetical order.
Scratch (reusable) variables are best grouped together and separated from the main variables by a blank line.

All variables are arrays that start with a single element. The command 'set the elements of' changes the number of elements assigned to the variable. If more are added, the existing contents are preserved. If the array is shortened, the highest-order values are lost.
Access to a given array element is done by setting an internal pointer: 'index <variable> to <index>'. The variable then behaves as if it has a single element. There is no other syntax available to access or write to elements of the variable.

AllSpeak is strongly typed, with one exception. The standard 'variable' type can hold numeric, string or boolean values. Conversion between types is often automatic but can be forced.
Where a variable has more than one element, they do not have to be identical types. Each element is independent of all the others.
There are significant differences between the JS and Python versions. The latter has 'dictionary' and 'list' to specifically handle data of those types. The JS version holds all content as strings and converts to and from objects as necessary.

## Collections
AllSpeak has 3 basic collection mechanisms for variables that hold string, numeric or boolean data:
1. Arrays are like cursors in SQL. They permit a single named variable to have multiple elements but only expose one at a time, so the array looks like a simple variable. Array elements do not have to be all of the same type.
2. Dictionaries are sets of key/value pairs. Any array element (as 1.) can itself be a dictionary. Dictionaries  use 'entry' as the primary access mechanism.
3. Lists are arrays of identically-typed values. Any array element (as 1.) can itself be a list. Lists use 'element' as the primary access mechanism.
In addition, any object type can have textual properties attached to it, as in
```
set property 'rank` of Button to `primary`
if property `rank` of Button is `primary` begin ... end
```

## Values
A value is a number, string or boolean entity; something that can be assigned to a general-purpose variable, which then adopts the entity as its value. Other types of variable may or may not have values. For example, a file does not. Each domain has its own set of value types with accompanying syntax and vocabulary.

## Conditions
A condition is something that is tested and returns either True or False. Each domain has its own set of condition types with accompanying syntax and vocabulary.

## Arithmetic
All arithmetic in AllSpeak is integer. If arithmetical operations are required on decimal items such as monetary values it is probably best to convert them to pence/cents for all the working.
Arithmetical operations follow natural language:
```
add A to B  
add A to B giving C  
take/subtract A from B  
take/subtrct A from B giving C  
multiply A by B  
multiply A by B giving C  
divide A by B  
divide A by B giving C
```
Remainders are dealt with by `A modulo B`

## Control flow
'fork' is used to park the current thread, start execution of the forked thread then resume the current thread. Because this is cooperative multitasking, threads do not run concurrently; each one yields when it reaches a 'stop' or 'wait' command.

## Modules
A module variable holds information about a script that has been loaded, compiled and run by its own script. The second script is an independent entity that can either run as a form of subroutine or in parallel with its parent.
Modules are useful when there is a substantial amount of code that performs a given function in a self-contained manner. Making it a module reduces the size of the parent script. A good rule of thumb is that once a script reaches 3000 lines it should probably be examined with a view to extracting one or more parts as modules.
Modules can import variables from their parent. Changes made to the variable by either module will be seen by the the other.
When a module is run it blocks its parent. Unblocking occurs if the module reaches 'exit', or it can explicitly 'release parent'. Once this happens, both scripts run concurrently using cooperative multitasking. When any script reaches 'stop' or 'wait', the runtime engine will resume processing of another script that is waiting (if any), creating the impression of true pre-emptive multitasking. Because this is not pre-emptive, commands can never be interrupted mid-way by another script.
A module can behave like a subroutine with private variables, by the use of messaging. A module can message its parent or its child, optionally passing a variable such as a dictionary. When the requested task has finished, the recipient then uses the same mechanism to return a result to its caller.

## Plugins
A plugin is a file that implements an extension to the vocabulary and/or syntax of AllSpeak, to handle specialized code usually written in JavaScript or Python and either included in the file or referenced by it. Plugins can bring the performance of an AllSpeak-based project very close to that of a conventionally-coded equivalent, while retaining the advantages of a scripted framework.

## Graphics
In the JS version, graphics are handled by the Browser module, which covers the majority of the DOM. SVG is handled by a plugin and there will possibly be others for jsGraph- or Canvas-based graphics.
The Browser module implements a wide range of variable types that each represent a DOM element such as a div, an input or an h1. These may be created as needed, but if a project is to comprise a UI of any size it is far better to describe its components using Webson, the companion UI language to AllSpeak. Webson represents HTML and CSS using a JSON dialect and provides an easy mapping between AllSpeak variables and DOM elements, allowing all visual elements to be described outside the AllSpeak scripts. This provides a clear separation of appearance and functionality.

## Strategy
This covers many things, so I'll add them as needed.
### When to use arrays
The best indicator is when several variables are seen to be doing much the same thing. Replacing them with one multi-element variable is usually the best choice. This can apply not just to string and numeric variables but also to other object types, most notably DOM elements.
### When to build a sub-module
When a script gets big - say around 2500 lines - it becomes hard to manage, even with the 'doc blocks' recently introduced. It will frequently be the case that one or more sections of the script are largely self-contained, with a lot of private variables and only limited interaction with the rest of the script. These are ideal candidates for consideration, especially if they run concurrently with the main script. Choose an optimal balance between exporting/importing variables at the point of launch and sending across data via internal messages.
### How to structure and use doc blocks
It's a matter of judgment how much or how little code to include in a block. The content should ideally be self-contained but this is not always possible.
Documentation in these blocks consists of a relatively small number of paragraphs. In normal editing mode they take up very little space since each paragraph is one line with no line breaks. In review mode they have immense value so they should prioritise being informative over being concise.
### When to build a plugin
There are occasions where large external functional blocks need to be called up. Plugins are the ideal mechanism; they provide a place to put the code and they cater for AllSpeak language extensions that make the code available for use by scripts. Examples are the MQTT module, which is now part of the core language, and Google Maps, which is already available as an external plugin.
### How to improve AllSpeak
If a task becomes awkward to construct using the available vocabulary/syntax and a viable alternative is spotted, feel free to put in a feature request at info@allspeak.ai
