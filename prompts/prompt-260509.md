## Doc blocks

This is experimental - an attempt to encourage code review by making it clearer and easier to follow. In particular, it aims to facilitate validation of vibe code by those who currently do not attempt to do so.

The idea is to divide scripts logically into functional blocks (this should not greatly affect the majority of scripts) and create documentation in the form of special comments that describe each block. The pattern, which I'll call 'doc blocks') is a number of special comment lines. Each one of these starts with !!, then an arbitrary number of spaces/tabs and then some text. After these lines there will be the code itself, then a line with just !! to terminate the doc block. Blank lines in the special comments can be created by using !! and a single space. These documentation lines will normally be written by AI and edited by the programmer.

A example is asedit.as.

The intention is to aid verification of code written by AI, by breaking it up into indiviually documented blocks that make it less intimidating, particularly for those who are not experts.

When code is heavily annotated with comments it can be harder to read and validate, not easier. So the editor will operate in 2 modes, selectable by a button next to Find. One mode is as at present, where every line is editable. I am anticipating that AI will usually doing the writing, so the extra text will not be a problem. The other mode will replace the page with a two-pane version (left and right with a draggable divider between them). The right-hand pane will show a doc block (with the !! and following spaces removed and the rest of the text reflowed to fit the pane width) and the left-hand pane will display the corresponding section of code. Up and down buttons at the top will move to the previous or next block.

Typically the very first doc block will describe in outline the entire script, and may have no associated code. After that, (ideally) every section of code will have a doc block to describe it in as much detail as is agreed between the agent and the programmer. This does not remove the need for conventional comments, which will be presented as normal.

Once the system is in use, any modification to existing code must trigger a check that the documentation still applies, and of course new code will always be created in a doc block.

Before starting, please comment on the practicality of doing this. We will create a new editor script so asedit.as will still be available unchanged. If you can think of any ways to improve the concept while retaining its core aims, these would be welcome.
