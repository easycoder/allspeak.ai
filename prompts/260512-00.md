# Curriculum review — analysis

Comparison and merge notes on the two earlier outlines:

- `260429-00.md` — original idioms outline (12 problem-oriented files under `idioms/`)
- `260429-01.md` — additions (13 concept-oriented reference sections)

## What's actually on the table

The two outlines aren't competing — they're at different altitudes.

- **260429-00** is *pattern-oriented*: "when X comes up, here's the AllSpeak way." Idioms.
- **260429-01** is *concept-oriented*: Symbols → Variables → Collections → Values → Conditions → Arithmetic → Control flow → Modules → Plugins → Graphics → Strategy. A language reference.

They don't merge into one tree because they answer different questions. A reader hitting "how do I handle multiple buttons?" wants idioms; a reader hitting "what *is* a dictionary in AllSpeak?" wants reference. Force them into one structure and both get worse.

## Proposal: two tiers under one root

```
learn/
  README.md            # which side to read first, and why
  reference/           # spine of the language
    01-structure.md
    02-symbols-and-layout.md
    03-variables-and-arrays.md
    ...
  idioms/              # patterns and worked examples
    01-cat-and-string-building.md
    02-event-handlers-and-array-index.md
    ...
```

Reference is canonical and stable; idioms cross-link into reference for "what is this thing" questions. Both ship in the repo, so AI agents and humans can pull whichever fits.

## Gaps

**In 260429-01 that should stay in reference:**

- The domain/plugin compiler model (§Structure) — load-bearing context for understanding how new vocabulary appears. Should be near the front.
- The 4-symbol rule, indenting, variable-naming conventions — boring but essential; AIs especially need this stated explicitly because they default to other-language conventions.
- Object properties (`set property … of …` / `if property …`) — easy to overlook, currently undocumented anywhere prominent.
- `fork` and cooperative multitasking semantics — the "never interrupted mid-command" guarantee changes how you write everything.
- Modules / messaging / `release parent` — the existing `as-modularize` skill is a good source to pull from but probably too narrow on its own.

**In 260429-00 that should stay in idioms:**

- `cat` placement rules (between, not before — single most common AI mistake).
- `or` / `on failure` semantics.
- Event handlers + `the index of` — *how does the handler know which array element fired* is the question every newcomer asks.
- REST async patterns; MQTT pub/sub conventions.
- Language directive + writing language-neutral logic.

**Neither outline covers, but should:**

- Iteration: `while`, `for each`, label loops — choices and when to pick which.
- String manipulation primitives (`replace`, `from … to …`, length-of, position-of).
- Date/time handling.
- `print` / `log` / debugging — workflow when something goes wrong.
- File I/O (mostly Python-side; flag the JS/Python divergence).
- Testing `.as` code — `conformance/` exists and is underused as a learning resource.
- The doc-block convention itself — lives in CLAUDE.md but deserves a learn/ entry because it's recent and least documented.

## Points in 260429-01 worth a second look

1. **"All arithmetic in AllSpeak is integer"** (§Arithmetic) — true for the engine, but the *floats-pass-through-as-backtick-strings; scale to ints for arithmetic* nuance should appear here, otherwise readers will be surprised when they see backtick floats in working scripts.
2. **JS-vs-Python collections** — phrasing implies Python is *better-typed*. That's only partly true and is in tension with the known Python i18n gaps (Python lags overall). Soften to "Python exposes dict/list as distinct types; JS unifies through stringification" without ranking them.
3. **§Modules** describes the runtime model well but doesn't mention *candidate-selection criteria* for when to extract — the `as-modularize` skill has worked-through examples (controller↔deviceControl, shell↔map-to-rooms) that should be cited here rather than duplicated.
4. **Symbol section** — `!!` and `!!!` (doc-block markers) aren't listed alongside `!`. Different lexical layer, but new readers will hit them in any modern script.

## Next step

Pick **one reference file** and **one idiom file** as paired prototypes — `reference/03-variables-and-arrays.md` + `idioms/02-event-handlers-and-array-index.md`. Drafting both at once stress-tests the cross-linking and surfaces whether the two-tier split actually pays off before committing for a dozen more.
