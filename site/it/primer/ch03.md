## Capitolo 3 — Flusso di Lavoro: Da Spazio Vuoto a MVP Affidabile

Questo capitolo fornisce un flusso di lavoro pratico passo-passo per costruire una webapp smartphone-first con assistenza AI.

È intenzionalmente agnostico rispetto al progetto: usatelo per qualsiasi app focalizzata dove l'apprendimento rapido e la consegna controllata contano.

### Posizionamento per nuovi programmatori (serio, non hype)

Questo flusso di lavoro è utile per le persone che entrano nello sviluppo software attraverso la pratica assistita dall'AI.

- L'AI sta cambiando il modo in cui il software viene costruito.
- I fondamentali e la revisione disciplinata contano ancora.
- Le iterazioni di progetto reali costruiscono sia competenze di programmazione che competenze di collaborazione con l'AI.

Evitare la narrativa del "successo istantaneo". Il progresso viene dalla comprensione di ogni iterazione.

### Vero punto di partenza: editor + agente AI + nessun file

Supponiamo che l'utente inizi solo con:

- un editor di codice,
- un agente di coding AI,
- uno spazio di lavoro vuoto.

Per rendere questo ripetibile, fornire una pagina primer user-friendly a un URL stabile.

Utilizzare due asset separati:

- URL della pagina marketing (rivolta agli umani)
- URL del primer per l'agente (markdown rivolto alla macchina)

Questa configurazione dovrebbe fare due cose:

1. Spiegare lo scopo del progetto e il metodo di lavoro.
2. Fornire un primo prompt che gli utenti possono incollare nel loro agente AI.

### Requisiti dell'URL del primer

Contenuto minimo nel primer per l'agente:

- posizioni del runtime e degli strumenti,
- convenzioni del progetto e ruoli dei file,
- comportamento di bootstrap atteso dall'agente,
- guida al testing locale per l'ambiente corrente.

### Comportamento atteso dell'agente al bootstrap

Dopo aver ricevuto il prompt iniziale, l'agente dovrebbe:

- spiegare un breve piano,
- creare i file minimi eseguibili,
- spiegare cosa fa ogni file,
- proporre e configurare il testing locale presto.

### Template del prompt iniziale

"Mi stai aiutando a costruire una webapp smartphone-first usando uno stack di scripting/runtime human-friendly.
Parti da uno spazio di lavoro vuoto.
Usa il primer per l'agente a <PRIMER_MD_URL> come contesto autorevole.

Spiega brevemente il tuo piano.
Fai il bootstrap del progetto dalle istruzioni del primer.
Poi spiega in linguaggio semplice cosa hai creato.

Solleva presto il testing locale nel browser, scegli la migliore opzione per questo ambiente e implementala se necessario."

### Sequenza URL del prompt per gli utenti

1. Aprire l'URL della pagina marketing.
2. Copiare il prompt iniziale (incluso l'URL del markdown del primer).
3. Incollarlo nell'agente AI.
4. Lasciare che l'agente faccia il bootstrap di file/strumenti dal primer.
5. Chiedere all'agente di spiegare cosa ha creato prima di procedere.

### Metodo a milestone (raccomandato)

Usare cicli brevi:

1. Definire un milestone ristretto.
2. Chiedere all'AI solo quel milestone.
3. Eseguire e verificare.
4. Revisionare con checklist.
5. Continuare.

### Blueprint dei milestone (agnostico rispetto al progetto)

#### M1 - Shell dell'app

- renderizzare il layout smartphone-first,
- inizializzare lo stato base,
- verificare il primo elemento interattivo.

#### M2 - Ciclo di interazione principale

- implementare l'azione utente primaria,
- collegare l'azione agli aggiornamenti di stato,
- mostrare il risultato visibile all'utente.

#### M3 - Arricchimento dati

- aggiungere una fonte dati esterna,
- gestire i percorsi di caricamento, successo e fallimento,
- assicurarsi che le risposte obsolete vengano ignorate.

#### M4 - Rafforzamento dell'affidabilità

- aggiungere protezioni contro azioni duplicate,
- migliorare il comportamento con rete lenta,
- aggiungere messaggi di retry e recupero.

### Vibe coding versus flusso di lavoro strutturato (confronto rapido)

**Stile vibe coding**
- veloce per l'esplorazione,
- tracciabilità debole,
- alto rischio di deviazione con la crescita dell'ambito.

**Stile flusso di lavoro strutturato**
- comunque veloce, ma con vincoli espliciti,
- migliore revisionabilità e riproducibilità,
- minor rischio di regressione durante la consegna sostenuta.

Usare il vibe coding per prototipi esplorativi grezzi.
Usare il flusso di lavoro strutturato per qualsiasi cosa che si prevede di mantenere.

### Regole pratiche per la collaborazione con l'AI

- Preferire molti piccoli prompt rispetto a uno enorme.
- Chiedere sempre all'AI di elencare i file modificati e i delta di comportamento.
- Chiedere all'AI di spiegare la logica non ovvia in linguaggio semplice.
- Bloccare l'accettazione se i revisori non comprendono una modifica.
- Mantenere lo slancio: prima chiarezza, poi velocità.

### Conclusione del capitolo

Il percorso affidabile più rapido è l'iterazione controllata:

- piccola modifica,
- risultato visibile,
- revisione esplicita,
- ripetere.

Questo è il punto di forza dello scripting human-friendly più la collaborazione AI: consegna ad alta velocità con logica leggibile e affidabilità pratica.
