## Capitolo 7 — Ingegneria dei Prompt per Team di Prodotto

I buoni prompt non riguardano formulazioni ingegnose.
Riguardano intento chiaro, vincoli espliciti e risultati testabili.

Questo capitolo fornisce un sistema di prompt pratico che i team possono riutilizzare tra le funzionalità.

### Perché la qualità dei prompt è importante

Prompt deboli creano output vaghi, rilavorazioni e rischio nascosto.
Prompt forti riducono l'ambiguità e aumentano la qualità dell'implementazione.

La qualità dei prompt influisce direttamente su:

- velocità di consegna,
- chiarezza del codice,
- tasso di difetti,
- fiducia del team.

### Anatomia del prompt (standard del team)

Utilizzare questa struttura per i prompt di implementazione:

1. **Risultato** — cosa deve essere vero dopo la modifica.
2. **Ambito** — cosa è incluso/escluso.
3. **Vincoli** — architettura, UX, stile e limiti degli strumenti.
4. **Accettazione** — come si verifica il successo.
5. **Note di rischio** — casi limite noti o aree fragili.

I prompt brevi vanno bene, purché questi elementi siano presenti.

### Template di prompt riutilizzabili

#### 1) Template per implementazione di funzionalità

"Implementa [funzionalità] per [schermata/flusso di lavoro].

Risultato:
- [comportamento atteso]

Ambito:
- Includere: [elementi]
- Escludere: [elementi]

Vincoli:
- Mantenere le modifiche piccole e focalizzate.
- Preservare il comportamento esistente al di fuori dell'ambito.
- Mantenere la UX smartphone-first.

Accettazione:
- [test 1]
- [test 2]

Report:
- file modificati,
- delta di comportamento,
- domande aperte."

#### 2) Template per correzione bug

"Correggi [sintomo del bug].

Riproduzione:
1. [passo]
2. [passo]

Atteso:
- [risultato corretto]

Vincoli:
- Trovare la causa radice.
- Evitare refactoring non correlati.

Accettazione:
- la riproduzione non fallisce più,
- il comportamento adiacente funziona ancora,
- riassumere perché questa correzione affronta la causa radice."

#### 3) Template per refactoring

"Refactoring di [area target] per [motivo: leggibilità/prestazioni/stabilità].

Non modificare il comportamento.
Mantenere le interfacce pubbliche invariate.

Accettazione:
- comportamento invariato,
- complessità ridotta,
- diff spiegato chiaramente."

### Anti-pattern dei prompt da evitare

- Obiettivi vaghi: "miglioralo."
- Ambito sovradimensionato in un solo passo.
- Vincoli mancanti.
- Nessun criterio di accettazione.
- Nessun requisito di spiegare il comportamento modificato.

Questi pattern aumentano la deviazione e riducono la fiducia.

### Quality gate per l'output AI

Prima di accettare una modifica generata dall'AI, richiedere:

- lista esplicita dei file modificati,
- spiegazione dell'impatto sul comportamento,
- evidenza di test o passi di verifica manuale,
- domande irrisolte segnalate chiaramente.

Se un gate fallisce, richiedere revisione prima dell'unione.

### Prompt di escalation (quando la qualità cala)

Utilizzare prompt di correzione diretta:

- "Restringi l'ambito solo a [X]."
- "Mostra la causa radice prima di proporre codice."
- "Elenca le alternative con compromessi in 3 punti."
- "Riesegui senza modifiche UX al di fuori di [area]."

I prompt di escalation recuperano il controllo rapidamente.

### Abitudine del team: libreria di prompt

Mantenere una libreria di prompt condivisa nel repository:

- template per funzionalità,
- template per bug,
- template per refactoring,
- prompt per checklist di rilascio,
- prompt per risposta agli incidenti.

Questo riduce la varianza nella qualità dei prompt tra i membri del team.

### Conclusione del capitolo

L'ingegneria dei prompt è una disciplina di consegna, non un trucco.
I team che standardizzano la struttura dei prompt producono codice più chiaro, meno regressioni e iterazione più rapida.
